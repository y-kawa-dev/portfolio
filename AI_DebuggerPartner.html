Overview

開発の背景と目的

抱えていた課題

  • • ダンプ解析の属人化: クラッシュダンプの解析ツール（WinDbg/CDB）は操作が難しく、対応できるエンジニアが限られていました。
  • • 対応の遅れ: エラー原因の調査やWeb検索を手動で行うため、解決策にたどり着くまでに時間がかかっていました。

解決策

  • デバッグの自動化: アプリが裏側でツールを制御し、ログ取得から解析までを自動実行。
  • 誰でも解析可能に: 難解なログをAIが分かりやすく要約・翻訳するため、経験の浅い人でも一次対応が可能に。

実装機能一覧

WinDbg連携 自動ダンプ解析

cdb.exeをバックグラウンドで起動し、Windowsダンプファイル（.dmp）を自動解析。

lm t n (ドライバ一覧) これらを一括実行し、AIが人間が読める形式で解説します。 エージェント的振る舞い: AIのプロンプトに「情報不足なら次のコマンドを人間に提案せよ」と指示を追加し、自律的なデバッグ支援を促進します。

マルチモーダル画像解析

ブルースクリーンやエラーダイアログの写真をVision AI（Gemini/Llama Vision）が直接「見て」原因を特定。OCRよりも高精度に状況を把握します。

2. ハイブリッドAIエンジン

Cloud & Local 切り替え

Gemini (Cloud): 高速・高精度な最新モデル。
Ollama (Local): 社外秘データやオフライン環境用のローカルLLMに対応。

モデル設定の柔軟性

設定画面からモデル名を自由に変更可能。将来のモデルアップデートにも即座に対応できる設計です。

3. インテリジェント・アシスト

正直な修復提案 (Honest AI)

物理故障（SSD破損等）の可能性がある場合は正直に「修復不能」と伝える安全設計。「なんでも直せる」と嘘をつきません。

スマートWeb検索

エラーコードは完全一致、文章は意味検索とクエリを自動最適化。検索先をMicrosoft公式等に限定し信頼性を担保します。

対話・WEB検索

解析結果の原因表示に加え、チャットで「分かりやすく教えて」等と聞くことでAIが対話形式でサポート。

さらに、アプリ右側のWEB検索ボタンをクリックすれば、AIが抽出した重要キーワード（エラーコード等）で即座にMicrosoft公式ドキュメントを検索。手動入力の手間なく裏付け調査を行えます。

調査・修復

「修復スクリプト」ボタンをクリックすると、AIが問題原因の解決法を自動判断し、目的に応じた修復スクリプトを自動で生成します。

生成されたスクリプトは、いきなり実行されず「ステップごとの確認（Enterキーで進行）」機能が付与された安全設計です。

ボタンが「スクリプトを保存」に変わったら、.ps1ファイルとしてダウンロード可能。あとは右クリックして実行するだけで、複雑な修復作業を自動化できます。

Technology Stack

採用技術とアーキテクチャ

Python & PySide6

非同期処理(QThread)による快適なGUIと、subprocessによる低レイヤー操作を両立。

Gemini API

高速な推論とマルチモーダル（画像認識）機能により、エラーログとスクリーンショットを同時に解析。

Ollama (Local)

Llama 3.2等のローカルLLMに対応。機密性の高いログを外部送信せずに解析するセキュアな運用を実現。

WinDbg (CDB)

Windows標準デバッガをバックエンドで制御。`!analyze -v` コマンドの結果を自動取得し、AIの入力へ変換。

開発を通じて学んだこと

「使いやすさ」が技術の価値を決める

高機能なAIや解析ロジックを実装しても、使い方が難しければ現場では使われないことを痛感しました。 そのため、「ドラッグ＆ドロップだけで開始する」「専門用語をAIに翻訳させる」「次に押すべきボタンを明確にする」といった、ユーザーの迷いを減らすUI設計に最も時間をかけました。

業務フロー全体を俯瞰する視点

単にエラー原因を表示するだけでなく、「その後の調査（Web検索）」や「解決（修復スクリプト）」、「報告（レポート作成）」までを一つのツール内で完結させることで、初めて実用的なツールになると学びました。 ユーザーが本当に求めているのは「解析結果」ではなく**「障害の解決」**であるという視点を持てたことが、今回の最大の収穫です。
